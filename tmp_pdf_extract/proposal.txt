User  Authentication  &  Login   The  system  shall  allow  users  to  log  in  using  valid  credentials   The  system  shall  allow  users  to  log  out   The  system  shall  allow  users  to  reset  their  password  using  email   The  system  shall  allow  users  to  change  their  password   The  system  shall  make  passwords  follow  password  creation  rules   The  system  shall  display  login  success  or  failure  messages  C.  Session  Handling   The  system  shall  automatically  log  out  users  after  a  period  of  inactivity   The  system  shall  request  user  to  enter  pin  to  resume  session  (Suggestion,  Need  Validation)   The  system  shall  restore  user’s  last  session  when  user  returns   The  system  shall  not  serve  2  instances  for  an  account  D.  Authorization  &  Roles   The  system  shall  allow  different  role  permissions  for  Admin  &  Expert   The  system  shall  assign  one  role  to  each  account   The  system  shall  restrict  access  to  Admin  features  to  Admin  accounts  only  E.  Admin  Dashboard   The  system  shall  allow  Admins  to  toggle  each  page’s  availability   The  system  shall  allow  Admins  to  upload  Outputs  to  Evaluate

 The  system  shall  allow  Admins  to  assign  an  Evaluation  to  specified  Experts   The  system  shall  allow  Admins  to  view  Evaluation  results   The  system  may  allow  Admins  to  export  Evaluation  results  F.  Expert  Evaluation   The  system  shall  show  Experts  a  list  of  Outputs  to  Evaluate   The  system  shall  show  Each  Evaluations’  Deadline  &  Completion  Status   The  system  shall  require  Experts  rate  a  Test  case  per  dimension   The  system  shall  minimize  and  hide  dimension  description  and  score  criteria  by  default,  but  can  be  expanded  to  be  shown   The  system  shall  allow  Experts  to  navigate  each  dimension   The  system  shall  only  allow  Experts  to  finish  an  evaluation  if  all  dimensions  are  rated   The  system  shall  allow  Experts  to  save  progress  and  continue  later  G.  Evaluation  Logic   The  system  shall  implement  Likert-scale  scoring  strictly  following  the  evaluation  dimensions  defined  in  the  research  paper.   The  system  shall  require  experts  to  rate  each  evaluation  dimension  before  submission.   The  system  shall  prevent  final  submission  if  any  dimension  remains  unrated.   The  system  shall  store  individual  dimension  scores  as  structured  numerical  values  to  support  aggregation  and  analysis.  H.  Evaluation  Notes  and  Qualitative  Feedback   The  system  shall  allow  experts  to  submit  optional  qualitative  notes  alongside  numerical  ratings.   The  system  shall  associate  notes  with:   The  corresponding  evaluation   The  specific  model  version   The  submitting  expert   The  system  shall  store  notes  in  a  flexible  data  structure  to  allow  varying  length  and  content.  I.  Model  Versioning   The  system  shall  associate  each  evaluation  with  exactly  one  model  version  identifier.   The  system  shall  prevent  modification  of  model  version  metadata  once  evaluations  have  been  submitted.   The  system  shall  allow  administrators  to  register  new  model  versions  for  future  evaluations.   The  system  shall  preserve  historical  evaluations  to  enable  cross-version  comparison.

Error  Handling   The  system  shall  automatically  log  errors  and  information    The  system  shall  notify  user  that  an  error  has  occurred  and  the  error  type   The  system  shall  attempt  to  continue  after  an  error  occurred  L. Tutorial  &  First  Login   The  system  shall  show  a  guided  tutorial  for  first-time  users   The  system  shall  show  a  pop-up  requiring  consent  to  collect  data  from  users   The  system  shall  request  a  new  password   N. III.  Non-Functional  Requirements  A.  Performance   The  system  shall  load  the  login  page  within  3  seconds  under  normal  network  conditions.   The  system  shall  load  dashboards  and  evaluation  pages  within  5  seconds  for  up  to  the  expected  concurrent  users.   The  system  shall  support  simultaneous  evaluations  by  multiple  experts  without  data  loss  or  noticeable  degradation.   Saving  evaluation  progress  shall  complete  within  2  seconds.   Notification  delivery  (in-app)  shall  occur  within  5  seconds  of  the  triggering  event.   Cached  data  (criteria,  model  versions,  summaries)  shall  be  retrievable  in  under  1  second.  B.  Scalability   The  system  shall  support  scaling  to  additional  expert  users  without  requiring  architectural  changes.   The  system  shall  support  an  increasing  number  of:  ●  Evaluations  ●  Model  versions   The  database  architecture  shall  support  growth  in  qualitative  notes  and  Likert-scale  responses  without  performance  degradation.  C.  Availability  &  Reliability   The  system  shall  maintain  an  availability  of  99%  uptime  excluding  scheduled  maintenance.   The  system  shall  gracefully  handle  partial  system  failures  without  data  corruption.   The  system  shall  ensure  that  submitted  evaluations  are  not  lost  due  to  system  crashes.   The  system  shall  allow  recovery  from  failures  using  backups.   Cached  data  shall  automatically  regenerate  if  cache  invalidation  occurs.  D.  Security  Requirements   The  system  shall  enforce  secure  authentication  using  encrypted  credentials.   The  system  shall  store  passwords  using  strong  hashing  algorithms.   The  system  shall  protect  against  common  web  vulnerabilities:  ●  SQL  Injection  ●  XSS

●  CSRF   The  system  shall  restrict  access  based  on  role  (Admin  vs  Expert).   The  system  shall  prevent  simultaneous  active  sessions  for  the  same  account.   The  system  shall  log  all  authentication  and  authorization  events.  E.  Data  Integrity  &  Consistency   The  system  shall  use  atomic  transactions  for  evaluation  submissions.   The  system  shall  prevent  incomplete  or  inconsistent  evaluation  records.   The  system  shall  ensure  each  evaluation  response  is  linked  to:  ●  Exactly  one  expert  ●  One  evaluation  output  ●  One  model  version   The  system  shall  prevent  modification  of  submitted  evaluations.   Historical  evaluation  data  shall  remain  immutable  once  finalized.  F.  Usability  &  Accessibility   The  system  shall  provide  a  responsive  UI  compatible  with  desktop,  tablet,  and  mobile  browsers.   The  system  shall  provide  clear  error  messages  understandable  by  non-technical  users.   The  system  shall  minimize  cognitive  load  by:  ●  Collapsing  dimension  descriptions  by  default  ●  Allowing  progressive  disclosure  of  information   The  system  shall  provide  a  guided  tutorial  for  first-time  users.   The  system  shall  support  keyboard  navigation  for  key  workflows.   The  system  shall  use  consistent  UI  elements  and  terminology  across  all  pages.  G.  Maintainability  &  Extensibility   The  system  shall  be  modular  to  allow  addition  of  new  features  (e.g.,  new  evaluation  dimensions).  (may  need  validation)   The  system  shall  allow  new  model  versions  to  be  added  without  impacting  evaluations  for  previous  models.   The  system  shall  log  system  errors  and  events  to  support  debugging.   The  system  shall  allow  configuration  changes  (e.g.,  notification  toggles)  without  redeployment.   The  codebase  shall  follow  consistent  coding  standards  and  documentation  practices.  (hahah  doubt)  H.  Compatibility   The  system  shall  support  modern  web  browsers  (Chrome,  Firefox,  Edge,  Safari).   The  system  shall  function  correctly  across  different  screen  resolutions.   The  system  shall  remain  compatible  with  future  database  schema  extensions.

Auditability  &  Logging   The  system  shall  log:  ●  Login  and  logout  events  ●  Evaluation  submissions  ●  Administrative  actions   Logs  shall  include  timestamps  and  user  identifiers. The  system  shall  preserve  message  ordering.   The  system  shall  support  concurrent  messaging  without  message  loss. Each  evaluation  response  shall  link:  
●  One  expert  user  ●  One  evaluation  output  ●  One  model  version
